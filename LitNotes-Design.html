Before the demo
  1. Create a Supabase project, paste URL + anon key in Settings → Collaboration (no rebuild needed for that step).
  2. (Optional) Configure Google OAuth and add the client IDs, then rebuild once for the new native Google Sign-In module.
  3. Run a two-device dry-run of live sharing (presence, cursors, edit/view roles) and a light/dark + permissions pass.
Confirmed decisions
  1. Theme follows the device default, defaults to Light when unset, with a manual System / Light / Dark override in Settings.
  2. Settings covers Appearance, Reading, Account, AI & API key, Collaboration, Permissions, Data & backup, About.
  3. Permissions (files, notifications, media for OCR) are requested during login and their grant-state is persisted.
  4. Sharing backend is Supabase Realtime; identity is Supabase Auth with Google + email, and guests can join a live room from an invite link.

9 · Live collaboration — how sharing works

A pluggable realtime layer on Supabase. The Share UI, presence, cursors and roles are identical regardless of backend; only the transport underneath is swappable.

Transport
One Supabase Realtime channel per room (room:<CODE>). WebSocket, free tier.
Presence
Who is in the room + name, colour and role — drives avatar pips and the participant list.
State sync
Canvas, threads, ink, highlights, index and OCR broadcast as debounced snapshots (last-writer-wins); late joiners request the current state.
Cursors
Broadcast in board/world coordinates, rendered per-peer and inverse-scaled to stay crisp at any zoom.
Roles
Owner / editor / viewer. Viewers are read-only (canvas editing + PDF highlighting disabled).
Invites
litnotes://join/<CODE>?a=edit|view deep link + 6-char room code; native share sheet.

10 · Sequence — live share session

1.Owner → CollabStore: startShare(edit|view)
2.CollabStore → Realtime: subscribe room:CODE + track presence
3.CollabStore → Store: getSyncState()
4.CollabStore → Realtime: broadcast snapshot
5.Peer → Realtime: join via link / code
6.Realtime → Peer: presence sync (avatars)
7.Peer → Realtime: request_snapshot
8.Realtime → Peer: snapshot → applyRemote
9.Owner → Store: edit card / ink
10.Store → Realtime: debounced snapshot
11.Realtime → Peer: live update + cursor

11 · Session state machine

Dashed transitions are error / return / logout edges. "Backgrounded / locked" is wired via AppState + SessionLockScreen (Settings → Account to disable).

Launching→ Unauthenticated→ Authenticating→ Permission prompt→ Onboarding→ Active session⇄ Backgrounded / locked

Authenticating → Unauthenticated on error · Active session → Unauthenticated on log out · Permission prompt → Active session for returning users.
